What does Cavekit's /ck:map command do?

Cavekit (by Julius Brussee, also author of Caveman) implements "spec is the product, code is the derivative." The four phases:

1. /ck:sketch — natural language → numbered requirements + acceptance criteria 2. /ck:map → dependency graph, tiered build plan (e.g., "34 tasks, 5 tiers") 3. /ck:make → parallel execution of independent tasks, validated at each tier gate 4. /ck:check → gap analysis, cross-model peer review (Codex optional)

/ck:sketch → "A REST API for tasks"
/ck:map   → "34 tasks, 5 tiers"  ← This step
/ck:make  → "CAVEKIT COMPLETE — 34 tasks, 18 iterations"


Key insight: The /ck:map step eliminates merge conflicts by grouping only truly independent tasks into each wave. This is a direct architectural solution to the compounding error problem — fewer sequential dependencies means shorter failure chains.

The Minimal Coding Agent Loop

The agentic loop in pseudocode:

async function agentLoop(task: string): Promise {
  const context: Message[] = [{ role: "user", content: task }];
  while (true) {
    const response = await claude.messages.create({ messages: context, tools });
    context.push({ role: "assistant", content: response.content });
    if (response.stop_reason === "end_turn") break;
    const toolResults = await executeTools(response.content);
    context.push({ role: "user", content: toolResults });
  }
}


Key insight: Any developer can build this loop in an afternoon. The engineering challenge is everything around it — what goes in the context, how you prompt the model, and how you prevent the agent from deleting your database.

The security lethal trifecta for AI agents

Simon Willison's "lethal trifecta" — the three conditions that make AI agents exploitable:

1. Access to private data — source code, API keys, secrets 2. Exposure to untrusted content — files the agent reads (prompt injection vectors) 3. Ability to externally communicate — network access for data exfiltration

Key insight: "Anyone who gets text into your LLM has full control over what tools it runs next." Breaking any ONE leg of the trifecta neutralizes the attack.

Most practical mitigation: Network isolation (HTTP proxy allowlisting, container sandboxing) — filesystem access and untrusted content are often unavoidable for useful work, but exfiltration channels can be cut.

Managed platforms () provide container sandboxing by default, making this an infrastructure concern rather than a developer burden.

Netflix's Temporal Buckets: Reliability Through Diversity

Netflix's video search architecture achieves agent reliability through an unexpected strategy: intentionally diversifying failure modes across modalities. Rather than one model that might fail in one systematic way, three parallel search paths (visual, audio, text) fail independently.

The temporal bucket fusion approach: - Divide video timeline into overlapping 30s buckets - Run visual/audio/semantic search on each bucket independently - Fuse results with reciprocal rank fusion — the highest agreement wins

// Reciprocal rank fusion: consensus beats any single signal
function fuseBuckets(results: BucketResult[][]): RankedResult[] {
  const scores = new Map();
  for (const rankedList of results) {
    rankedList.forEach((r, rank) => {
      scores.set(r.id, (scores.get(r.id) ?? 0) + 1 / (60 + rank));
    });
  }
  return [...scores.entries()]
    .sort(([, a], [, b]) => b - a)
    .map(([id]) => ({ id }));
}


Key insight: Reliability doesn't require a single perfect model — it requires that independent failure modes cancel each other out.

What Is the Role of raw/ in the Wiki Ingest Pipeline?

The ingest pipeline has two strictly separated layers:

1. raw/ — immutable archived sources. One file per URL, with frontmatter metadata. Never modified or deleted. 2. wiki/ — compiled knowledge. Synthesized from many sources. Updated, merged, and enriched over time.

A source fans out to 2-5 wiki chapters. A wiki chapter draws from 2+ sources. The 1:1 mapping (one source → one wiki page) is explicitly forbidden — that's just a bookmark with extra steps.

Key insight: The raw→wiki split enforces separation of evidence from synthesis. You can always re-derive the wiki from raw sources. You cannot reconstruct raw sources from the wiki.

Why Svelte Can Target Native Apps (React Can't As Easily)

React and Svelte both support multi-platform rendering, but for fundamentally different reasons:

React is a runtime library. It achieves multi-platform (React Native) by splitting its reconciler from its renderer — the same runtime runs in both environments, just with different renderers. Bundle size travels with every target.

Svelte is a compiler. It generates platform-specific code at build time with near-zero runtime. Custom renderers work by splitting the compilation output into an intermediate representation (fragment tree) that any target renderer can consume.

| Aspect | React Native | Svelte Custom Renderers | |--------|-------------|------------------------| | Approach | Runtime split | Compiler IR | | Bundle overhead | ~45KB React runtime | Near zero | | Status | Production (2015) | POC/research (2025–26) | | Reactivity model | VDOM diff at runtime | Compiled state machines |

Key insight: Svelte's compiler foundation isn't just a performance win for the web — it's an architectural advantage for multi-platform. When you compile to a platform-specific IR instead of carrying a runtime, you get smaller bundles and faster startup everywhere, not just in the browser.

What does Rating.Again do in FSRS?

The four FSRS ratings map to different memory states:

import { Rating } from 'ts-fsrs';
// Rating.Again → "I couldn't recall this at all"  → tomorrow
// Rating.Hard  → "I struggled significantly"       → 2-3 days
// Rating.Good  → "I recalled with effort"          → 3-7 days  
// Rating.Easy  → "I recalled instantly"            → 7+ days


Each successful review multiplies the spacing interval — Again breaks the chain and resets it.

Key insight: The FSRS algorithm optimizes for a ~90% recall rate at review time. "Again" tells it you're well below that threshold — so it brings the card back immediately. Fighting the urge to rate Hard when you should rate Again leads to a buildup of poorly-learned cards that feel like they're in your queue but aren't actually retained.
