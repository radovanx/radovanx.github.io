LLM-as-Judge: Scaling Evaluation Without Human Labelers

// LLM-as-judge: a cheap model evaluates expensive model outputs at scale
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

async function llmJudge(task: string, output: string): Promise<{
  score: number; pass: boolean; reasoning: string;
}> {
  const { content } = await client.messages.create({
    model: "claude-haiku-4-5-20251001",  // cheap model for high-volume evaluation
    max_tokens: 256,
    messages: [{
      role: "user",
      content: Task: ${task}\nOutput: ${output}\n\nJSON only: {"score":1-5,"pass":bool,"reasoning":"one sentence"},
    }],
  });
  const text = content[0].type === "text" ? content[0].text : "{}";
  return JSON.parse(text);
  // { score: 4, pass: true, reasoning: "Accurate but omits edge case handling." }
}

// Self-consistency hallucination detection: high variance = likely hallucinating
async function detectHallucination(question: string, attempts = 3): Promise {
  const answers = await Promise.all(
    Array.from({ length: attempts }, () =>
      client.messages.create({
        model: "claude-haiku-4-5-20251001",
        max_tokens: 64,
        messages: [{ role: "user", content: question }],
      }).then(r => (r.content[0].type === "text" ? r.content[0].text : "").slice(0, 40))
    )
  );
  return new Set(answers).size > 1; // true = inconsistent = likely hallucinating
}


Key insight: LLM-as-judge separates "demo AI" from "production AI." Without evaluation loops, quality degrades as the world changes. With them, each cycle improves: user interactions → implicit/explicit signals → curate examples → update model → A/B test → deploy.

LLM Wiki Pattern vs. GraphRAG: same idea, different outputs

Naive RAG:   query → retrieve chunks → synthesize (every query)
GraphRAG:    precompute graph → query → graph-guided synthesis (faster)
LLM Wiki:    precompute wiki pages → read page (no inference at query time)

The Chain Method: Why 5 Focused Prompts Beat 1 Complex Prompt

Never ask an LLM to do 5 things in one prompt. Chain them — each focused step compounds quality, and you can review and correct between stages.

Prompt 3: "Based on these gaps: [paste]

Prompt 4: "Based on this positioning: [paste]

Why it works: Each step has a shorter failure chain. Quality compounds because each input is already validated. Correction is possible at each stage — impossible with a monolithic prompt.

Key insight: Monolithic prompts ask for 5 things → the weakest step contaminates all downstream steps. The chain method isolates failures so they can be fixed before they compound.

Why Svelte Can Target Native Apps (React Can't As Easily)

React and Svelte both support multi-platform rendering, but for fundamentally different reasons:

React is a runtime library. It achieves multi-platform (React Native) by splitting its reconciler from its renderer — the same runtime runs in both environments, just with different renderers. Bundle size travels with every target.

Svelte is a compiler. It generates platform-specific code at build time with near-zero runtime. Custom renderers work by splitting the compilation output into an intermediate representation (fragment tree) that any target renderer can consume.

| Aspect | React Native | Svelte Custom Renderers | |--------|-------------|------------------------| | Approach | Runtime split | Compiler IR | | Bundle overhead | ~45KB React runtime | Near zero | | Status | Production (2015) | POC/research (2025–26) | | Reactivity model | VDOM diff at runtime | Compiled state machines |

Key insight: Svelte's compiler foundation isn't just a performance win for the web — it's an architectural advantage for multi-platform. When you compile to a platform-specific IR instead of carrying a runtime, you get smaller bundles and faster startup everywhere, not just in the browser.
