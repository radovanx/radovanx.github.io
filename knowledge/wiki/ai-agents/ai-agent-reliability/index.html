AI Agent Reliability — From Failure Modes to Working Patterns

AI agents are simultaneously capable of superhuman feats and fundamentally unreliable at multi-step reasoning. This page compiles everything we know about why they fail and which architectural patterns actually make them work.

Neural network processing and failure modes

The Fundamental Limit: Compounding Error

Vishal Sikka (former SAP CTO, studied under AI founder John McCarthy) and his son Varin published research — covered by Futurism — claiming to mathematically demonstrate that LLMs cannot perform computational tasks beyond relatively low complexity thresholds.

The math is straightforward probability theory. If each step succeeds with probability p, a chain of n steps succeeds with probability p^n:

Even at 95% accuracy per step:

  • 10-step task → 0.95^10 = 60% success rate
  • 50-step task → 0.95^50 = 7.7% success rate
  • 100-step task → 0.95^100 = 0.6% success rate
# Compounding error rate demonstration
# Shows how per-step accuracy decays over multi-step agent tasks

def success_rate(per_step_accuracy: float, num_steps: int) -> float:
    """Calculate end-to-end success rate for a multi-step agent task.
    
    This is the fundamental reliability equation for autonomous agents:
    P(success) = accuracy^steps
    
    Args:
        per_step_accuracy: Probability of each step succeeding (0.0 to 1.0)
        num_steps: Number of sequential steps in the task
    
    Returns:
        End-to-end probability of the full task succeeding
    """
    return per_step_accuracy ** num_steps


# Per-step accuracies to compare
accuracies = [0.95, 0.98, 0.99]
step_counts = [5, 10, 25, 50, 100]

print(f"{'Steps':<8}", end="")
for acc in accuracies:
    print(f"{'p=' + str(acc):<12}", end="")
print()
print("-" * 44)

for steps in step_counts:
    print(f"{steps:<8}", end="")
    for acc in accuracies:
        rate = success_rate(acc, steps)
        print(f"{rate:.1%}{'':>5}", end="")
    print()

# Output:

# 10      59.9%       81.7%       90.4%
# 25      27.7%       60.3%       77.8%
# 50       7.7%       36.4%       60.5%
# 100      0.6%       13.3%       36.6%

Each step multiplies the probability of failure. This compounding error rate is the hard ceiling on autonomous agent reliability. OpenAI has acknowledged that hallucinations persist despite advances and model accuracy will never reach 100%.

Key insight: Even at 95% per-step accuracy, a 100-step task succeeds less than 1% of the time. Even at 99% per-step accuracy, a 100-step task succeeds only 37% of the time. Reliability is not a model problem — it is a systems architecture problem.

Companies that downsized to replace workers with AI agents discovered they couldn’t sustain the same output quality — the compounding error problem hits hardest in sustained, multi-step workflows.

Hallucination: The Core Failure Mode

When models generate plausible but false information, every downstream decision is undermined. This isn’t a bug to be fixed — it’s an emergent property of statistical pattern completion. The model doesn’t retrieve knowledge; it completes patterns, and sometimes the most probable completion is wrong.

The Paradox: Superhuman AND Unreliable

Anthropic’s Project Glasswing deploys AI to find security vulnerabilities that escaped 27 years of human review and 5 million automated test runs. The same technology can’t reliably do multi-step math.

Resolution: AI excels at pattern matching at scale (scanning millions of code paths for vulnerability signatures) but fails at sequential reasoning (chaining dependent logic steps where each builds on the last).

Key insight: Don’t ask agents to reason through 50 sequential steps. Ask them to match patterns across massive search spaces. Design for breadth, not depth.

This distinction is the key to designing reliable agent systems.

No single approach solves the problem. The practical path: combine multiple strategies and design systems that tolerate failures. Sikka acknowledged that external guardrails and supporting components around models can overcome inherent LLM weaknesses. The model alone isn’t enough, but the system around it can be.

Automation and reliability testing patterns

Four Patterns That Actually Work

Every reliable AI system we’ve studied uses one or more of these architectural patterns. They share a single meta-insight: move work out of the unreliable real-time LLM call and into reliable pre-computed infrastructure.

Anthropic’s own Building Effective Agents guide validates this approach, distinguishing between workflows (predefined code paths) and agents (dynamic, self-directed processes). Their core recommendation: “Start simple, measure rigorously, increase complexity only when justified by measurable performance gains.” The patterns below align with Anthropic’s design principles — simplicity, transparency, and investing in tool quality (the “Agent-Computer Interface”).

Pattern 1: Knowledge Compilation (Not RAG)

Instead of asking an LLM to derive answers from raw data each time (unreliable), pre-compile knowledge into structured artifacts where errors are visible and correctable.

  • The LLM wiki pattern pre-compiles knowledge into cross-referenced pages
  • Glasswing pre-compiles vulnerability signatures into scanning patterns
  • This is why wikis beat chatbots for knowledge work — synthesis is durable, not ephemeral

The key advantage: compilation surfaces errors. A wiki page with a wrong claim is visible and fixable. A chatbot hallucination in a one-off response is invisible and lost.

The pattern was articulated by Andrej Karpathy in his LLM Wiki concept: instead of retrieving from raw documents and reasoning in real-time (RAG), maintain a “persistent, compounding artifact” where the LLM pre-synthesizes knowledge into cross-referenced pages. As Karpathy describes it, the wiki “automatically updates cross-references, keeps summaries current” — shifting the maintenance burden from humans to LLMs while making errors visible and correctable.

"""
Knowledge Compilation vs RAG: Reliability Comparison

RAG re-derives answers from raw sources every time → each derivation is
a new opportunity for hallucination, and errors are ephemeral (invisible).

Knowledge compilation pre-synthesizes into durable artifacts → errors are
visible, correctable, and compound into knowledge rather than failure.

Reference: Karpathy's LLM Wiki pattern
https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
"""

from dataclasses import dataclass, field
from datetime import date


@dataclass
class WikiPage:
    """A compiled knowledge page — draws from multiple sources,
    cross-references other pages, and makes claims explicit and reviewable."""
    title: str
    content: str
    sources: list[str] = field(default_factory=list)
    cross_references: list[str] = field(default_factory=list)
    updated: date = field(default_factory=date.today)
    claims: list[str] = field(default_factory=list)


@dataclass
class RawSource:
    """An immutable source document — the LLM reads but never modifies."""
    url: str
    title: str
    content: str
    saved: date = field(default_factory=date.today)


class RAGApproach:
    """Retrieve-and-generate: re-derive answers from raw sources each time.
    
    Failure mode: each query is a fresh opportunity for hallucination.
    Errors are ephemeral — a wrong answer in chat disappears after the session.
    """
    
    def __init__(self, sources: list[RawSource]):
        self.sources = sources
    
    def query(self, question: str) -> str:
        # Step 1: Retrieve relevant sources (simulated)
        relevant = [s for s in self.sources if question.lower().split()[0] in s.content.lower()]
        
        # Step 2: LLM derives answer from raw sources — NEW derivation every time
        # This is where hallucination happens: the model must synthesize
        # across sources in real-time, with no prior compilation to anchor it
        return f"[Derived from {len(relevant)} sources — may hallucinate]"


class KnowledgeCompilationApproach:
    """Pre-compile knowledge into reviewed, cross-referenced artifacts.
    
    Key advantages over RAG:
    1. Errors surface in persistent pages (visible, fixable)
    2. Cross-references are pre-computed (no real-time reasoning needed)
    3. Contradictions are flagged at compile time (not query time)
    4. Each compilation compounds on previous work (not starting from zero)
    """
    
    def __init__(self):
        self.wiki: dict[str, WikiPage] = {}
        self.sources: list[RawSource] = []
    
    def ingest(self, source: RawSource) -> list[str]:
        """Compile a new source into the wiki — update existing pages,
        add cross-references, flag contradictions.
        
        A single source should update 5-15 existing pages.
        Creating a new page is the exception, not the rule.
        """
        self.sources.append(source)
        pages_updated = []
        
        # For each existing page, check if this source adds knowledge
        for title, page in self.wiki.items():
            if self._is_relevant(source, page):
                page.sources.append(source.url)
                page.updated = date.today()
                pages_updated.append(title)
        
        return pages_updated
    
    def query(self, question: str) -> str:
        # Query reads pre-compiled wiki — no real-time derivation needed
        # Answers are grounded in reviewed, cross-referenced pages
        relevant_pages = [p for p in self.wiki.values()
                          if question.lower().split()[0] in p.content.lower()]
        return f"[Compiled from {len(relevant_pages)} wiki pages — reviewed, cross-referenced]"
    
    def _is_relevant(self, source: RawSource, page: WikiPage) -> bool:
        """Check if a source has knowledge relevant to this wiki page."""
        return any(term in source.content.lower() 
                   for term in page.title.lower().split())

Key insight: Knowledge compilation shifts errors from invisible (ephemeral chat responses) to visible (persistent artifacts that can be reviewed and corrected over time).

Pattern 2: Dynamic Tool Search (Not Static Tool Loading)

As agents gain access to more tools, a scaling problem emerges: loading all tool definitions into the context window becomes impractical and degrades accuracy.

The problem:

  • 50 tools consume 10-20K tokens just for definitions
  • Tool selection accuracy degrades beyond 30-50 loaded tools
  • Large catalogs (hundreds or thousands) are unusable without search

The solution: Tool definitions are withheld from context. The agent receives a summary of available capabilities and searches for relevant tools on demand. Only 3-5 relevant tools load per task and remain available for subsequent turns.

Claude’s Agent SDK implements this via ENABLE_TOOL_SEARCH:

ValueBehavior
true (default)Always on — definitions never loaded upfront
autoActivates when definitions exceed 10% of context
auto:NCustom threshold (e.g., auto:5 for 5%)
falseOff — all tools loaded every turn

Limits: Maximum 10,000 tools in catalog. Returns 3-5 most relevant per search. Haiku models don’t support it.

Why it improves reliability: Fewer irrelevant tools in context means fewer wrong tool calls. The agent isn’t distracted by 47 tools it doesn’t need.

Optimization: Tool names and descriptions drive search quality. search_slack_messages surfaces better than query_slack. A system prompt listing available categories helps the agent know what to search for.

The following example demonstrates the tool search pattern conceptually — how an agent dynamically discovers and loads tools from a large catalog instead of carrying all definitions in context:

"""
Dynamic Tool Search Pattern

Instead of loading all tool definitions into the LLM context upfront,
this pattern maintains a searchable catalog and loads only relevant
tools per task. This mirrors how Claude's Agent SDK implements
ENABLE_TOOL_SEARCH.

Reference: https://platform.claude.com/docs/en/agent-sdk/tool-search
"""

from dataclasses import dataclass
from typing import Callable


@dataclass
class ToolDefinition:
    """A tool that an agent can use. Name and description drive search quality."""
    name: str
    description: str
    parameters: dict
    handler: Callable


class ToolSearchCatalog:
    """Searchable catalog of tools — definitions stay OUT of LLM context
    until explicitly loaded by a search query.
    
    Key insight: at 50+ tools, putting all definitions in context degrades
    tool selection accuracy. Search keeps context lean and selection precise.
    """
    
    def __init__(self, tools: list[ToolDefinition], max_results: int = 5):
        self.tools = {t.name: t for t in tools}
        self.max_results = max_results
    
    def search(self, query: str) -> list[ToolDefinition]:
        """Find relevant tools by matching query against names and descriptions.
        
        In production, this would use embedding similarity or BM25.
        Claude's SDK returns 3-5 most relevant tools per search.
        """
        scored = []
        query_terms = query.lower().split()
        for tool in self.tools.values():
            searchable = f"{tool.name} {tool.description}".lower()
            score = sum(1 for term in query_terms if term in searchable)
            if score > 0:
                scored.append((score, tool))
        scored.sort(key=lambda x: x[0], reverse=True)
        return [tool for _, tool in scored[:self.max_results]]
    
    def get_summary(self) -> str:
        """One-line summary of available capabilities — this IS loaded into
        context so the agent knows what to search for."""
        categories = set()
        for tool in self.tools.values():
            # Extract category from tool name (e.g., "slack" from "search_slack_messages")
            parts = tool.name.split("_")
            if len(parts) > 1:
                categories.add(parts[1] if parts[0] in ("search", "get", "create", "update", "delete") else parts[0])
        return f"Available tool categories: {', '.join(sorted(categories))}"


class AgentWithToolSearch:
    """Agent that searches for tools on demand instead of loading all upfront.
    
    Context budget comparison:
    - 50 tools loaded statically ≈ 10,000-20,000 tokens
    - Tool search summary ≈ 200 tokens + 3-5 tools loaded on demand ≈ 1,500 tokens
    """
    
    def __init__(self, catalog: ToolSearchCatalog):
        self.catalog = catalog
        self.loaded_tools: dict[str, ToolDefinition] = {}
    
    def process_task(self, task: str) -> dict:
        """Process a task by searching for and loading relevant tools."""
        # Step 1: Search catalog for relevant tools
        relevant_tools = self.catalog.search(task)
        
        # Step 2: Load only the relevant tools into working context
        for tool in relevant_tools:
            self.loaded_tools[tool.name] = tool
        
        # Step 3: Agent now has only 3-5 relevant tools, not 500
        return {
            "loaded_tools": [t.name for t in relevant_tools],
            "context_tokens_saved": "~18,000 (vs loading all 50+ tools)",
            "catalog_summary": self.catalog.get_summary(),
        }


# Example usage:
tools = [
    ToolDefinition("search_slack_messages", "Search Slack messages by keyword, channel, or user", {}, lambda: None),
    ToolDefinition("create_jira_ticket", "Create a new Jira issue with title, description, and priority", {}, lambda: None),
    ToolDefinition("get_github_pr", "Fetch pull request details, diff, and review comments", {}, lambda: None),
    ToolDefinition("search_confluence_pages", "Search Confluence wiki pages by title or content", {}, lambda: None),
    ToolDefinition("send_email_notification", "Send an email notification to a user or distribution list", {}, lambda: None),
    # ... imagine 500 more tools here
]

catalog = ToolSearchCatalog(tools)
agent = AgentWithToolSearch(catalog)

result = agent.process_task("find the slack discussion about the authentication bug")
# result["loaded_tools"] → ["search_slack_messages"]
# Only 1 tool loaded into context instead of 500+

For production use with Anthropic’s Claude API, tool search is configured via the ENABLE_TOOL_SEARCH environment variable on the Agent SDK. The SDK handles the search infrastructure automatically — you provide tools with good names and descriptions, and the SDK handles discovery.

Pattern 3: Persistent Memory (Not Stateless Sessions)

Without persistent memory, agents re-derive context on every interaction. Users repeat themselves. The agent can’t learn preferences, track projects, or build on previous work.

Architecture patterns:

  • File-based memory — structured knowledge in files the agent reads on startup (e.g., CLAUDE.md, memory directories). Simple, transparent, version-controllable.
  • Database-backed — conversation summaries, user preferences, learned facts in a database. Better for high-volume structured data.
  • Vector stores — embed and retrieve relevant past interactions semantically. Good for fuzzy recall across large histories.
  • Hybrid — structured files for stable knowledge + vector search for dynamic recall. Most practical for real systems.

Real-world applications:

  • Stock market analysis agents that track portfolio history and market insights across sessions
  • Development agents that remember codebase architecture and user preferences (Claude Code’s memory system)
  • Research agents that compound knowledge over time (this wiki)

Why it improves reliability: Instead of re-deriving answers (error-prone), the agent verifies against accumulated, previously-validated knowledge. Each session builds on the last instead of starting from zero.

Pattern 4: Managed Agent Infrastructure (Not DIY Harnesses)

Anthropic’s Managed Agents (beta, 2026) represent a new architectural pattern: instead of building your own agent loop, you use managed cloud infrastructure that handles orchestration, sandboxing, and session persistence. See claude managed agents for full architecture details.

Core abstractions:

  • Agent — Configuration (model + system prompt + tools + MCP servers). Defines WHAT.
  • Environment — Cloud container with isolated execution. Defines WHERE.
  • Session — Running instance connecting agent to environment. Defines WHEN/HOW.
  • Events — Messages, tool calls, and results flowing through sessions. The protocol.

Built-in tools: Bash, file operations, web search, and MCP server connections — all sandboxed in the container.

Why it improves reliability: The infrastructure layer handles the failure-prone parts of agent systems:

  • Container isolation prevents tool execution from affecting other processes
  • Session persistence maintains state without custom memory implementations
  • Event streaming provides a clean protocol for agent-user communication
  • Rate limiting and safety guardrails are built in (60 creates/min, 600 reads/min)

Research preview features hint at where this is headed: Outcomes (defining success criteria), Multiagent (multiple agents in shared environments), and Memory (cross-session persistence). These directly address the compounding error problem — outcomes provide verification checkpoints, multiagent enables specialization (reducing per-agent step count), and memory enables the Pattern 3 persistence without custom implementation.

The analogy: This is to agent development what cloud computing was to server management. You stop building the harness and start configuring the agent. The reliability burden shifts from the developer to the platform.

The open alternative: LangChain’s Deep Agents Deploy (beta, April 2026) offers the same pattern — managed harness with sandboxes, sessions, and 30+ endpoints — but model-agnostic and open-source (MIT). Their key argument: memory lock-in matters more than model lock-in. When your agent’s accumulated knowledge is stored behind a proprietary API, switching platforms means losing context. Deep Agents stores memory in standard formats (AGENTS.md, skills files) that users own and can self-host. See claude managed agents for the detailed comparison.

The market is splitting: integrated platforms (Anthropic — battle-tested, locked-in) vs. open harnesses (LangChain — flexible, newer). Both validate Pattern 4’s core insight: stop building harnesses, start configuring agents. The disagreement is about who controls the infrastructure.

This Wiki as Case Study

This project embodies all four patterns:

  • Knowledge compilation: Raw sources → synthesized wiki pages with cross-references. Errors are visible in the compiled pages, not hidden in ephemeral chat responses.
  • Persistent memory: The wiki itself IS the memory — each session builds on previous work, each ingest compounds on existing knowledge.
  • Tool search: As the wiki grows, load only relevant pages for each query instead of the full index.
  • Managed infrastructure: The wiki runs on Claude Code, which handles session management, file access, and tool orchestration — the same shift that Managed Agents brings to the broader agent ecosystem.

The wiki pattern reframes the reliability problem from “can the LLM reason correctly right now?” to “has the LLM compiled correct knowledge over time?” — a much easier problem because errors compound into visibility rather than compounding into failure.

Key insight: The meta-pattern across all four approaches is the same: move work out of the unreliable real-time LLM call and into reliable pre-computed infrastructure.

Multimodal AI: When Modalities Compound

Netflix’s video search system (Synchronizing the Senses, April 2026, by Meenakshi Jindal and Munya Marazanye) demonstrates a fifth reliability strategy: multimodal fusion. Instead of relying on a single AI modality (text, vision, or audio), their system orchestrates an ensemble of specialized models — character recognition, scene detection, dialogue analysis, embedding generation — to search production footage. Each modality compensates for the others’ weaknesses.

This is the same principle as knowledge compilation applied to perception: rather than trusting a single noisy signal, synthesize multiple signals into a more reliable composite. The compounding error problem still applies (each modality has its own failure modes), but the correlation between failures is low — visual hallucinations rarely coincide with audio hallucinations on the same clip.

Key insight: Multimodal AI doesn’t eliminate unreliability — it diversifies it. When failure modes are uncorrelated, combining modalities produces reliability that exceeds any single modality.

The Scale Problem

A standard 2,000-hour production archive contains over 216 million frames. When processed through an ensemble of specialized models, this baseline explodes into billions of multi-layered data points. The metadata varies wildly — from discrete text labels to dense vector embeddings — and must be synchronized into a unified chronological map while maintaining sub-second query latency.

Netflix’s answer is a three-stage decoupled pipeline that separates concerns for resilience:

StagePurposeTechnologyReliability Mechanism
1. Transactional PersistenceSafely capture every model outputApache CassandraData integrity + high write throughput
2. Offline Data FusionFuse overlapping annotations from different modelsApache Kafka → batch jobsDecoupled from intake — heavy computation never bottlenecks ingestion
3. Real-Time IndexingEnable sub-second cross-annotation queriesElasticsearch (nested docs)Upsert via composite key (asset ID + time bucket) — single source of truth

Temporal Bucket Fusion

The key architectural innovation is fixed-size temporal bucketing — discretizing continuous model outputs into one-second intervals, then intersecting them:

  1. Bucket Mapping — A character detected from seconds 2-8 maps into seven distinct one-second buckets
  2. Annotation Intersection — When “Joey” (character model) and “kitchen” (scene model) overlap at second 4, they fuse into a single record
  3. Optimized Persistence — Enriched records written back as second-by-second multi-modal index

This is the same pattern as knowledge compilation at the infrastructure level: instead of reasoning across raw model outputs at query time (unreliable), pre-compute the intersections so queries just look up fused records (reliable).

/**
 * Temporal Bucket Fusion — Netflix's pattern for synchronizing
 * multimodal AI model outputs into a unified search index.
 *
 * The key insight: discretize continuous detections into fixed-size
 * time buckets, then intersect across models. Queries hit pre-fused
 * records instead of reasoning across raw outputs at runtime.
 *
 * Reference: Netflix Technology Blog, "Synchronizing the Senses" (April 2026)
 */

interface Annotation {
  annotationId: string;
  type: "CHARACTER_SEARCH" | "SCENE_SEARCH" | "DIALOGUE_SEARCH";
  label: string;
  startTimeNs: number;
  endTimeNs: number;
  embeddingVector?: number[];
  confidenceScore?: number;
}

interface TemporalBucket {
  assetId: string;
  bucketStartNs: number;
  bucketEndNs: number;
  sourceAnnotations: Annotation[];
}

const BUCKET_SIZE_NS = 1_000_000_000; // 1-second buckets

function mapToBuckets(
  assetId: string,
  annotation: Annotation
): TemporalBucket[] {
  const buckets: TemporalBucket[] = [];
  let start = annotation.startTimeNs;

  // Discretize continuous detection into fixed-size intervals
  while (start < annotation.endTimeNs) {
    const bucketStart =
      Math.floor(start / BUCKET_SIZE_NS) * BUCKET_SIZE_NS;
    buckets.push({
      assetId,
      bucketStartNs: bucketStart,
      bucketEndNs: bucketStart + BUCKET_SIZE_NS,
      sourceAnnotations: [annotation],
    });
    start = bucketStart + BUCKET_SIZE_NS;
  }
  return buckets;
  // "Joey" detected 2s-8s → 7 one-second buckets
  // Each bucket holds a reference to the original annotation
}

function fuseAnnotations(
  existing: TemporalBucket,
  incoming: TemporalBucket
): TemporalBucket {
  // When multiple models annotate the same time bucket,
  // fuse into a single comprehensive record
  return {
    ...existing,
    sourceAnnotations: [
      ...existing.sourceAnnotations,
      ...incoming.sourceAnnotations,
    ],
  };
  // "Joey" + "kitchen" at second 4 → one fused record
  // Queries can now find "Joey in the kitchen" without
  // cross-referencing separate model outputs at runtime
}


<?php
/**
 * Temporal Bucket Fusion — Netflix's pattern for synchronizing
 * multimodal AI model outputs into a unified search index.
 *
 * Reference: Netflix Technology Blog, "Synchronizing the Senses" (April 2026)
 */

class Annotation {
    public function __construct(
        public readonly string $annotationId,
        public readonly string $type, // CHARACTER_SEARCH, SCENE_SEARCH, DIALOGUE_SEARCH
        public readonly string $label,
        public readonly int $startTimeNs,
        public readonly int $endTimeNs,
        public readonly ?array $embeddingVector = null,
        public readonly ?float $confidenceScore = null,
    ) {}
}

class TemporalBucket {
    /** @param Annotation[] $sourceAnnotations */
    public function __construct(
        public readonly string $assetId,
        public readonly int $bucketStartNs,
        public readonly int $bucketEndNs,
        public array $sourceAnnotations = [],
    ) {}
}

const BUCKET_SIZE_NS = 1_000_000_000; // 1-second buckets

/** @return TemporalBucket[] */
function mapToBuckets(string $assetId, Annotation $annotation): array
{
    $buckets = [];
    $start = $annotation->startTimeNs;

    while ($start < $annotation->endTimeNs) {
        $bucketStart = intdiv($start, BUCKET_SIZE_NS) * BUCKET_SIZE_NS;
        $buckets[] = new TemporalBucket(
            assetId: $assetId,
            bucketStartNs: $bucketStart,
            bucketEndNs: $bucketStart + BUCKET_SIZE_NS,
            sourceAnnotations: [$annotation],
        );
        $start = $bucketStart + BUCKET_SIZE_NS;
    }
    return $buckets;
    // "Joey" detected 2s-8s → 7 one-second buckets
}

function fuseAnnotations(
    TemporalBucket $existing,
    TemporalBucket $incoming,
): TemporalBucket {
    return new TemporalBucket(
        assetId: $existing->assetId,
        bucketStartNs: $existing->bucketStartNs,
        bucketEndNs: $existing->bucketEndNs,
        sourceAnnotations: array_merge(
            $existing->sourceAnnotations,
            $incoming->sourceAnnotations,
        ),
    );
    // "Joey" + "kitchen" at second 4 → one fused record
}


Hybrid Search: Symbolic + Semantic


The query engine combines text matching with vector search — neither alone is reliable enough:


    

  • Exact vs. Approximate Vector Search — Toggle between exact k-NN (uncompromising precision) and ANN via HNSW (speed at massive scale). Different workflows need different trade-offs.
    • 

  • Dynamic Similarity Metrics — Cosine similarity and Euclidean distance, swappable per model because different training architectures shape vector spaces differently.
    • 

  • Phrase Matching with Slop — match-phrase queries with configurable proximity tolerance. “Friends don’t lie” finds the correct Stranger Things scene even if the user’s memory slightly deviates from the exact transcription.
    • 

  • N-Gram Search-as-You-Type — Edge N-gram tokenizers index dialogue substrings for frame-accurate results as editors type.
    • 

  • Levenshtein Fuzzy Matching — Accounts for transcription anomalies and phonetic misspellings.
    • 

  • Result Curation — Post-processing reconstructs original time boundaries using union (full span, prioritizes breadth) or intersection (exact overlap, prioritizes co-occurrence) logic.
    • 



This hybrid approach maps directly to the meta-pattern: symbolic text matching catches what vector search misses (exact names, specific phrases), and vector search catches what text matching misses (semantic similarity, visual context). The combination is more reliable than either alone.

Pattern 5: Security Sandboxing (The Lethal Trifecta)


Simon Willison’s Living Dangerously with Claude talk (Oct 2025) identifies a critical security dimension to agent reliability. Claude Code’s --dangerously-skip-permissions (“YOLO mode”) enables genuinely transformative autonomous work — Willison completed three full projects in 48 hours — but creates a lethal trifecta when three conditions combine:


    

  1. Access to private data — source code, API keys, secrets
    2. 

  2. Exposure to untrusted content — files the agent reads (potential prompt injection vectors)
    3. 

  3. Ability to externally communicate — network access for data exfiltration
    4. 





Key insight: “Anyone who gets text into your LLM has full control over what tools it runs next.” Prompt injection through untrusted files can steal credentials via the agent’s own tool use.




The solution is sandboxing, specifically network isolation:


    

  • Run agents on managed infrastructure (Claude Code for web, OpenAI Codex Cloud)
    • 

  • HTTP proxy allowlisting to control outbound network access
    • 

  • macOS sandbox-exec for policy-based process isolation
    • 

  • Container isolation (Docker, managed environments)
    • 



Breaking any one leg of the trifecta neutralizes the attack. Network control is the most practical — filesystem access and untrusted content are often unavoidable for useful work, but exfiltration channels can be cut.


This connects directly to Pattern 4 (Managed Infrastructure): platforms like Managed Agents and Deep Agents Deploy provide container sandboxing by default, making the security problem an infrastructure concern rather than a developer burden.

TDD as AI Verification (Beck’s Resurgence)


Kent Beck — who pioneered Test-Driven Development — argues TDD is becoming “more relevant than ever for working with AI” (Pragmatic Summit, April 2026). AI-generated code requires stronger verification practices, not weaker ones. This maps directly to the meta-pattern: move trust from the unreliable generation step to reliable pre-computed verification infrastructure (tests).


Martin Fowler adds a critical observation: AI performs well on greenfield projects but “doesn’t work nearly as well on large and complex codebases.” This is the compounding error problem manifesting at the codebase level — the more context an agent must hold, the more steps it chains, the lower its reliability.


Their joint insight on team structure: two humans plus AI agents outperform solo developers with agents. The “re-soloing” effect — one person managing ten agents — loses the social complexity that historically produced quality. Fowler predicts two-pizza teams (5-10 people) will become more capable with agents as force multipliers, not shrink to one-pizza teams.




Key insight: AI doesn’t replace teams — it amplifies them. The re-soloing trap (one engineer + many agents) loses the collaborative judgment that catches errors no test suite covers.




The Builder Mindset


Building reliable AI systems against these failure statistics requires a specific disposition: the willingness to keep building despite hard limits. self belief strategic optimism frames this as “strategic optimism” — conviction that architectural patterns (not raw model capability) will eventually yield reliable systems. The engineers behind tools like Glasswing and Claude’s Agent SDK kept investing despite knowing the compounding error math. The pattern works because they designed around it, not in spite of it.


Fowler and Beck’s disruption cycles framework reinforces this: every major shift (microprocessor, internet, OO, Agile) followed the same pattern — initial hype, snake oil, organizational confusion, and eventually the practices that stuck. The AI shift is no different, except “there is no argument about how important” it is. The builders who thrive will be those who maintain Fowler’s balance of healthy skepticism (decades of snake oil) with genuine curiosity — the same strategic optimism applied to an industry-wide transition.


Pattern 6: Engineering Quality Gates (Agent Skills)


The reliability patterns above address infrastructure — architecture, tooling, sandboxing. Agent Skills (Addy Osmani, 13k+ stars, v0.5.0 April 2026) addresses the process layer: enforcing the engineering discipline that human teams apply but AI agents skip by default.


The core problem: AI coding agents are capable but undisciplined. Left unconstrained, they skip specs, bypass tests, ignore security reviews, and optimize for “done” over “correct.” Each shortcut is a small-probability failure that compounds through the task.


Agent Skills encodes 20 engineering workflows + 7 commands into Markdown skill files that agents load before starting work:










































CommandPhaseWhat It Prevents
/specDefineBuilding the wrong thing
/planPlanUnderestimating dependencies, skipping edge cases
/buildBuildOver-engineering, scope creep
/testVerifyUntested behavior becoming load-bearing
/reviewReviewSecurity holes, perf regressions, tech debt
/shipShipPremature release, missing rollback plan


The framework encodes Google’s engineering culture — Hyrum’s Law (any observable behavior becomes load-bearing), Chesterton’s Fence (understand before deleting), Shift Left (catch defects early) — as anti-rationalization safeguards agents can’t talk themselves out of.


Key innovation: Source-Driven Development. The newest skill directly targets hallucination at the code level: “Ground every framework decision in official documentation — verify, cite sources, flag what’s unverified.” Agents frequently invent API parameters, use deprecated methods, or pattern-complete from stale training data. Source-driven development forces the agent to fetch current docs and cite them before coding — the same principle as knowledge compilation applied to individual coding decisions.




Key insight: Architecture patterns (knowledge compilation, tool search, sandboxing) reduce infrastructure-level failure. Engineering quality gates (Agent Skills) reduce process-level failure. Both are necessary — an agent that follows a rigorous spec-before-code workflow but runs in an unsandboxed environment is still vulnerable; an agent that runs on robust infrastructure but writes code without testing it still ships bugs.




The skills ecosystem is growing into a category: caveman (token compression), make-interfaces-feel-better (design quality), agent-skills (engineering quality) — all of them Markdown files that override LLM defaults with opinionated, specific rules.
