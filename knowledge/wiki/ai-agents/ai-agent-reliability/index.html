AI Agent Reliability — From Failure Modes to Working Patterns

AI agents are simultaneously capable of superhuman feats and fundamentally unreliable at multi-step reasoning. This page compiles everything we know about why they fail and which architectural patterns actually make them work.

The Fundamental Limit: Compounding Error

Vishal Sikka (former SAP CTO, studied under AI founder John McCarthy) and his son Varin published research claiming to mathematically demonstrate that LLMs cannot perform computational tasks beyond relatively low complexity thresholds.

Even at 95% accuracy per step:

  • 10-step task → 60% success rate
  • 50-step task → 7.7% success rate
  • 100-step task → 0.6% success rate

Each step multiplies the probability of failure. This compounding error rate is the hard ceiling on autonomous agent reliability. OpenAI has acknowledged that hallucinations persist despite advances and model accuracy will never reach 100%.

Companies that downsized to replace workers with AI agents discovered they couldn’t sustain the same output quality — the compounding error problem hits hardest in sustained, multi-step workflows.

Hallucination: The Core Failure Mode

When models generate plausible but false information, every downstream decision is undermined. This isn’t a bug to be fixed — it’s an emergent property of statistical pattern completion. The model doesn’t retrieve knowledge; it completes patterns, and sometimes the most probable completion is wrong.

The Paradox: Superhuman AND Unreliable

Anthropic’s [[ai-security/glasswing-project|Project Glasswing]] deploys AI to find security vulnerabilities that escaped 27 years of human review and 5 million automated test runs. The same technology can’t reliably do multi-step math.

Resolution: AI excels at pattern matching at scale (scanning millions of code paths for vulnerability signatures) but fails at sequential reasoning (chaining dependent logic steps where each builds on the last).

This distinction is the key to designing reliable agent systems. Don’t ask agents to reason through 50 sequential steps. Ask them to match patterns across massive search spaces.

No single approach solves the problem. The practical path: combine multiple strategies and design systems that tolerate failures. Sikka acknowledged that external guardrails and supporting components around models can overcome inherent LLM weaknesses. The model alone isn’t enough, but the system around it can be.

Three Patterns That Actually Work

Every reliable AI system we’ve studied uses one or more of these architectural patterns. They share a single meta-insight: move work out of the unreliable real-time LLM call and into reliable pre-computed infrastructure.

Pattern 1: Knowledge Compilation (Not RAG)

Instead of asking an LLM to derive answers from raw data each time (unreliable), pre-compile knowledge into structured artifacts where errors are visible and correctable.

  • The [[knowledge-management/llm-wiki-pattern|LLM wiki pattern]] pre-compiles knowledge into cross-referenced pages
  • Glasswing pre-compiles vulnerability signatures into scanning patterns
  • This is why wikis beat chatbots for knowledge work — synthesis is durable, not ephemeral

The key advantage: compilation surfaces errors. A wiki page with a wrong claim is visible and fixable. A chatbot hallucination in a one-off response is invisible and lost.

Pattern 2: Dynamic Tool Search (Not Static Tool Loading)

As agents gain access to more tools, a scaling problem emerges: loading all tool definitions into the context window becomes impractical and degrades accuracy.

The problem:

  • 50 tools consume 10-20K tokens just for definitions
  • Tool selection accuracy degrades beyond 30-50 loaded tools
  • Large catalogs (hundreds or thousands) are unusable without search

The solution: Tool definitions are withheld from context. The agent receives a summary of available capabilities and searches for relevant tools on demand. Only 3-5 relevant tools load per task and remain available for subsequent turns.

Claude’s Agent SDK implements this via ENABLE_TOOL_SEARCH:

ValueBehavior
true (default)Always on — definitions never loaded upfront
autoActivates when definitions exceed 10% of context
auto:NCustom threshold (e.g., auto:5 for 5%)
falseOff — all tools loaded every turn

Limits: Maximum 10,000 tools in catalog. Returns 3-5 most relevant per search. Haiku models don’t support it.

Why it improves reliability: Fewer irrelevant tools in context means fewer wrong tool calls. The agent isn’t distracted by 47 tools it doesn’t need.

Optimization: Tool names and descriptions drive search quality. search_slack_messages surfaces better than query_slack. A system prompt listing available categories helps the agent know what to search for.

Pattern 3: Persistent Memory (Not Stateless Sessions)

Without persistent memory, agents re-derive context on every interaction. Users repeat themselves. The agent can’t learn preferences, track projects, or build on previous work.

Architecture patterns:

  • File-based memory — structured knowledge in files the agent reads on startup (e.g., CLAUDE.md, memory directories). Simple, transparent, version-controllable.
  • Database-backed — conversation summaries, user preferences, learned facts in a database. Better for high-volume structured data.
  • Vector stores — embed and retrieve relevant past interactions semantically. Good for fuzzy recall across large histories.
  • Hybrid — structured files for stable knowledge + vector search for dynamic recall. Most practical for real systems.

Real-world applications:

  • Stock market analysis agents that track portfolio history and market insights across sessions
  • Development agents that remember codebase architecture and user preferences (Claude Code’s memory system)
  • Research agents that compound knowledge over time (this wiki)

Why it improves reliability: Instead of re-deriving answers (error-prone), the agent verifies against accumulated, previously-validated knowledge. Each session builds on the last instead of starting from zero.

This Wiki as Case Study

This project embodies all three patterns:

  • Knowledge compilation: Raw sources → synthesized wiki pages with cross-references. Errors are visible in the compiled pages, not hidden in ephemeral chat responses.
  • Persistent memory: The wiki itself IS the memory — each session builds on previous work, each ingest compounds on existing knowledge.
  • Future: tool search: As the wiki grows, load only relevant pages for each query instead of the full index.

The wiki pattern reframes the reliability problem from “can the LLM reason correctly right now?” to “has the LLM compiled correct knowledge over time?” — a much easier problem because errors compound into visibility rather than compounding into failure.

The Builder Mindset

Building reliable AI systems against these failure statistics requires a specific disposition: the willingness to keep building despite hard limits. [[productivity/self-belief-strategic-optimism]] frames this as “strategic optimism” — conviction that architectural patterns (not raw model capability) will eventually yield reliable systems. The engineers behind tools like Glasswing and Claude’s Agent SDK kept investing despite knowing the compounding error math. The pattern works because they designed around it, not in spite of it.
