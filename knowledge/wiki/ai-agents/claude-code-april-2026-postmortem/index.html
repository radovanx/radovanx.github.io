Claude Code April 2026 Postmortem — Three Bugs, One Month

Sources

  • An update on recent Claude Code quality reports (Anthropic Engineering, April 2026) — official postmortem detailing three separate bugs that caused quality degradation, what was found, fixed, and process changes

Between March and April 2026, Anthropic shipped three independent changes that combined to produce widespread reports of Claude Code quality degradation. Each bug affected different users on different timelines — making the aggregate look like broad, inconsistent decline. All three were resolved by April 20, 2026 (v2.1.116).

Claude Code postmortem — three separate changes, one month of degradation reports

Key insight: None of these three bugs were the same bug. They overlapped in time, affected different traffic slices, and individually looked like normal variation. Together they looked like systemic degradation. The lesson: independent changes can compound into correlation that masks root cause.

Bug 1 — Reasoning Effort Default Changed (March 4)

Anthropic changed Claude Code’s default reasoning effort from high to medium after Opus 4.6 launched with occasional very long tail latencies in high mode — the UI appeared frozen.

The internal reasoning: In evals, medium effort achieved slightly lower intelligence with significantly less latency for most tasks, better usage limit headroom, and no long-tail latency spikes.

What happened: Users reported Claude Code felt less intelligent. Anthropic shipped design iterations — effort selector UI, ultrathink re-enabled, startup notices — but most users kept the medium default without knowing what had changed.

Resolution (April 7): Reverted. Current defaults:

  • xhigh for Opus 4.7
  • high for all other models

Architecture note: Effort levels map to positions on the test-time-compute curve. The effort parameter is sent to the Messages API; users can override via /effort. The tradeoff is genuine — more thinking → better output, but at latency and token cost.

Bug 2 — Caching Optimization That Dropped Prior Reasoning (March 26)

This is the most technically interesting bug. When Claude reasons through a task, thinking blocks are stored in conversation history so Claude can see why it made prior edits and tool calls on every subsequent turn.

Intended design: When a session is idle for >1 hour (cache eviction threshold), clear old thinking sections once using the clear_thinking_20251015 API header + keep:1. Resume sending full reasoning history afterward.

The bug: Instead of clearing once, the flag cleared thinking on every subsequent turn for the rest of the session. After the idle threshold was crossed once, each turn kept only the most recent thinking block and discarded everything before it.

Compounding failure: If a follow-up message arrived while Claude was mid-tool-use, it started a new turn under the broken flag — even the current turn’s reasoning was dropped. Claude continued executing, but increasingly without memory of why it had chosen to do what it was doing.

Observed symptoms:

  • Forgetfulness and repetitive behavior
  • Odd tool choices
  • Usage limits draining faster than expected (dropped thinking = cache misses)

Why it was hard to catch: The bug only manifested in stale sessions (>1 hour idle). Two unrelated experiments — an internal server-side queue change and a separate change in how thinking was displayed — masked the bug in most internal CLI sessions. It passed multiple code reviews, unit tests, E2E tests, automated verification, and dogfooding.

A notable finding: When Anthropic back-tested the offending pull requests with Code Review, Opus 4.7 (with full repository context) found the bug. Opus 4.6 didn’t. This directly led to shipping support for additional repositories as context for code reviews.

Fixed April 10, v2.1.101

clear_thinking_20251015 header + keep:1
↓
Intended: clears once when session resumes after cache eviction
Bug:      clears on EVERY turn after the first idle-session resume

Bug 3 — Verbosity System Prompt Change (April 16)

With Opus 4.7 being notably more verbose than its predecessor, Anthropic added a verbosity constraint to the Claude Code system prompt during pre-launch tuning:

"Length limits: keep text between tool calls to ≤25 words. Keep final responses to ≤100 words unless the task requires more detail."

What they tested: Multiple weeks of internal testing with the eval set in use at the time — no regressions observed.

What they missed: Additional ablation testing with a broader eval suite showed a 3% drop for both Opus 4.6 and Opus 4.7. The constraint was too aggressive — it traded off against coding quality.

Reverted April 20, v2.1.116

Key insight: System prompt changes to AI coding agents interact with model behavior in non-obvious ways. A verbosity constraint that looks like a formatting preference can affect reasoning quality — the model suppresses intermediate thinking that would otherwise improve output.

Compounding Effect

Each change in isolation:

  • Bug 1: Affected reasoning quality. Measurable but ambiguous.
  • Bug 2: Affected session continuity. Intermittent (only idle sessions).
  • Bug 3: Affected coding quality. Broad but subtle (3% on evals).

Together: Different users were hit by different combinations at different times. The aggregate looked like chronic, inconsistent degradation across all of Claude Code — but no single monitor showed a clean signal because each bug’s blast radius was different.

Process Changes

Anthropic committed to several changes after the investigation:

AreaChange
DogfoodingMore internal staff using exact public build, not feature-test builds
Code ReviewAdditional repository context support; shipping improved version to customers
System prompt governancePer-model evals for every system prompt change; ablations on every line; new audit tooling
CLAUDE.mdGuidance added: model-specific changes gated to the specific model they target
Intelligence tradeoffsSoak periods, broader eval suite, gradual rollouts for any change that could affect intelligence
Transparency@ClaudeDevs on X for in-depth product decision explanations; centralized GitHub threads

Usage limits were reset for all subscribers on April 23.

Lessons for Anyone Building on Claude Code

On reasoning effort: The effort parameter is not just a latency knob — it has material quality implications. high/xhigh is the right default for coding; medium works for simple tasks. The /effort command lets users tune this per-session.

At the Messages API level, reasoning effort maps to thinking.budget_tokens:

import Anthropic from "@anthropic-ai/sdk";
const client = new Anthropic();

// "high" effort ≈ large thinking budget — quality difference is real, not cosmetic
async function reviewWithEffort(code: string, effort: "high" | "medium"): Promise<string> {
  const budgetTokens = effort === "high" ? 10_000 : 2_000;
  const res = await client.messages.create({
    model: "claude-opus-4-7",
    max_tokens: 16_000,
    thinking: { type: "enabled", budget_tokens: budgetTokens },
    messages: [{ role: "user", content: `Review for bugs:\n\n${code}` }],
  });
  const text = res.content.find(b => b.type === "text");
  return text?.type === "text" ? text.text : "";
  // high: slower, finds subtle bugs; medium: faster, may miss edge cases
  // measure on YOUR eval set before defaulting to medium
}


<?php
// PHP: set thinking budget explicitly — don't assume defaults are "high"
function reviewWithEffort(string $code, string $effort = 'high'): string {
    $budgetTokens = $effort === 'high' ? 10_000 : 2_000;
    $ch = curl_init('https://api.anthropic.com/v1/messages');
    curl_setopt_array($ch, [
        CURLOPT_POST           => true,
        CURLOPT_POSTFIELDS     => json_encode([
            'model'      => 'claude-opus-4-7',
            'max_tokens' => 16_000,
            'thinking'   => ['type' => 'enabled', 'budget_tokens' => $budgetTokens],
            'messages'   => [['role' => 'user', 'content' => "Review for bugs:\n\n{$code}"]],
        ]),
        CURLOPT_HTTPHEADER     => [
            'x-api-key: ' . getenv('ANTHROPIC_API_KEY'),
            'anthropic-version: 2023-06-01',
            'content-type: application/json',
        ],
        CURLOPT_RETURNTRANSFER => true,
    ]);
    $body = json_decode(curl_exec($ch) ?: '{}', true);
    curl_close($ch);
    foreach ($body['content'] ?? [] as $block) {
        if ($block['type'] === 'text') return $block['text'];
    }
    return '';
}


On thinking blocks: Prior reasoning in conversation history is load-bearing — Claude uses it to remember why it made decisions. APIs that touch thinking history (like clear_thinking_20251015) require careful semantics: once vs. per-turn is a catastrophic difference in a long session.


On system prompt verbosity constraints: Explicit word limits (≤25 words between tool calls) are too coarse — they can suppress reasoning. Prefer structural verbosity controls (model training, UX patterns) over raw token budgets in prompts.


// ❌ WRONG — hard word limits suppress reasoning chains, not just filler prose
const FRAGILE_SYSTEM_PROMPT = `Keep text between tool calls to ≤25 words.`;

// ✅ CORRECT — structural guidance without a word budget
const ROBUST_SYSTEM_PROMPT = `Be concise. Skip preamble and filler. Focus on the task.`;
// The 3% quality drop came from the ≤25 word limit, not from "be concise"
