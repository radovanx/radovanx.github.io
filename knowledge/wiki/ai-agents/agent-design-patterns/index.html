Agent Design Patterns — Four Architectures That Actually Work

Every reliable AI system uses one or more of these patterns. They share a meta-insight: move work out of the unreliable real-time LLM call and into reliable pre-computed infrastructure.

Pattern 1: Knowledge Compilation (Not RAG)

Instead of asking an LLM to derive answers from raw data each time (unreliable), pre-compile knowledge into structured artifacts where errors are visible and correctable.

  • The LLM wiki pattern pre-compiles knowledge into cross-referenced pages
  • Glasswing pre-compiles vulnerability signatures into scanning patterns
  • Wikis beat chatbots for knowledge work — synthesis is durable, not ephemeral

The key advantage: compilation surfaces errors. A wiki page with a wrong claim is visible and fixable. A chatbot hallucination in a one-off response is invisible and lost.

"""
Knowledge Compilation vs RAG — Reliability Comparison

RAG re-derives answers from raw sources every time → each derivation
is a fresh opportunity for hallucination (errors are ephemeral, invisible).

Knowledge compilation pre-synthesizes into durable artifacts → errors are
visible, correctable, and compound into knowledge rather than failure.
"""

from dataclasses import dataclass, field
from datetime import date


@dataclass
class WikiPage:
    """A compiled knowledge page — multiple sources, cross-references, reviewable claims."""
    title: str
    content: str
    sources: list[str] = field(default_factory=list)
    cross_references: list[str] = field(default_factory=list)
    updated: date = field(default_factory=date.today)


class KnowledgeCompilationApproach:
    """Pre-compile knowledge into reviewed, cross-referenced artifacts.
    
    Key advantages over RAG:
    1. Errors surface in persistent pages (visible, fixable)
    2. Cross-references are pre-computed (no real-time reasoning needed)
    3. Contradictions are flagged at compile time (not query time)
    4. Each compilation compounds on previous work (not starting from zero)
    """
    
    def __init__(self):
        self.wiki: dict[str, WikiPage] = {}
    
    def ingest(self, source_url: str, source_content: str) -> list[str]:
        """Compile a new source into the wiki — updates existing pages,
        adds cross-references, flags contradictions.
        
        A single source should update 2-5 existing pages.
        """
        pages_updated = []
        for title, page in self.wiki.items():
            if any(term in source_content.lower() for term in title.lower().split()):
                page.sources.append(source_url)
                page.updated = date.today()
                pages_updated.append(title)
        return pages_updated

The TypeScript equivalent — same fan-out logic, same compounding advantage over RAG:

// TypeScript equivalent: knowledge compilation — one source fans out across many pages
interface WikiPage {
  title: string;
  content: string;
  sources: string[];
  crossReferences: string[];
  updated: string; // ISO date "YYYY-MM-DD"
}

class KnowledgeBase {
  private wiki = new Map<string, WikiPage>();

  ingest(sourceUrl: string, sourceContent: string): string[] {
    const pagesUpdated: string[] = [];
    for (const [title, page] of this.wiki) {
      const terms = title.toLowerCase().split(" ");
      if (terms.some(term => sourceContent.toLowerCase().includes(term))) {
        page.sources.push(sourceUrl);
        page.updated = new Date().toISOString().slice(0, 10);
        pagesUpdated.push(title);
      }
    }
    return pagesUpdated;
    // Output: ["AI Agents", "Tool Search"] — one source updates multiple pages
    // vs RAG: re-derives the same connections on every query
  }
}


The PHP version — useful for CMS-backed wikis and WordPress integrations:


<?php
class KnowledgeBase {
    private array $wiki = [];

    public function ingest(string $sourceUrl, string $sourceContent): array {
        $pagesUpdated = [];
        foreach ($this->wiki as $title => &$page) {
            $terms = explode(' ', strtolower($title));
            foreach ($terms as $term) {
                if (str_contains(strtolower($sourceContent), $term)) {
                    $page['sources'][] = $sourceUrl;
                    $page['updated']   = date('Y-m-d');
                    $pagesUpdated[]    = $title;
                    break;
                }
            }
        }
        return $pagesUpdated;
        // Output: ["AI Agents", "Tool Search"] — errors visible in persistent pages, not hidden in chat
    }
}




Key insight: Knowledge compilation shifts errors from invisible (ephemeral chat responses) to visible (persistent artifacts that can be reviewed and corrected over time).




Pattern 2: Dynamic Tool Search (Not Static Tool Loading)


As agents gain more tools, loading all definitions into context becomes impractical and degrades accuracy.























ProblemSolution
50 tools = 10-20K tokens of definitionsWithhold definitions — agent gets a summary instead
Tool selection accuracy degrades beyond 30-50 toolsAgent searches for relevant tools on demand
1000-tool catalogs are unusableOnly 3-5 relevant tools load per task


Claude’s Agent SDK implements this via ENABLE_TOOL_SEARCH:



























ValueBehavior
true (default)Always on — definitions never loaded upfront
autoActivates when definitions exceed 10% of context
auto:NCustom threshold (e.g., auto:5 for 5%)
falseOff — all tools loaded every turn

Optimization: Tool names and descriptions drive search quality. search_slack_messages surfaces better than query_slack. A system prompt listing available categories helps the agent know what to search for.


Limits: Max 10,000 tools. Returns 3-5 most relevant per search. Haiku models don’t support it.


// TypeScript: dynamic tool search vs. static loading
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

// 50 tools × ~300 tokens = ~15,000 tokens loaded per request (static)
const TOOL_CATALOG: Anthropic.Tool[] = [
  {
    name: "search_slack_messages", // specific name → surfaces in "search Slack" queries
    description: "Search Slack messages by keyword, channel, or date range",
    input_schema: { type: "object", properties: { query: { type: "string" }, channel: { type: "string" } }, required: ["query"] },
  },
  {
    name: "query_github_issues",
    description: "Query GitHub issues by repo, label, state, or author",
    input_schema: { type: "object", properties: { repo: { type: "string" }, label: { type: "string" } }, required: ["repo"] },
  },
  // ... 48 more tools (~15K tokens total when loaded statically)
];

// ❌ Static: all definitions in every request context
const static_response = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 4096,
  tools: TOOL_CATALOG, // 15K tokens wasted when only 1-2 tools are relevant
  messages: [{ role: "user", content: "Search Slack for the API design discussion" }],
});

// ✅ Dynamic (Claude Agent SDK: ENABLE_TOOL_SEARCH=true, the default)
// Tool summaries stay in context; full definitions fetched only for relevant tools
const agent = await client.beta.agents.create({
  model: "claude-sonnet-4-6",
  tools: TOOL_CATALOG,
  system: "Manage team workspace. Available categories: slack, github, confluence, jira.",
  // ENABLE_TOOL_SEARCH=true by default — loads 3-5 definitions per task, not all 50
});
// "search_slack_messages" → found for "search Slack" queries (~300 tokens loaded)
// "query_slack"           → might not surface — generic name loses semantic signal
// Output: only slack search tools loaded; 50× fewer tokens than static approach


<?php

$toolCatalog = [

// System prompt hint: list categories so the agent knows what to search for
$systemPrompt = "Manage team workspace.\n\nAvailable tool categories:\n" .

    "- Tasks: jira, linear\n\nSearch for tools by category before using them.";

// versus loading all 50 definitions (~15K tokens) on every request.


Pattern 3: Persistent Memory (Not Stateless Sessions)


Without persistent memory, agents re-derive context on every interaction. Users repeat themselves. The agent can’t learn preferences or build on previous work.


graph TD
    subgraph Stateless["❌ Stateless — Re-derive every time"]
        S1["Session 1\n(derive context)"] -.-> S2["Session 2\n(re-derive context)"]
        S2 -.-> S3["Session 3\n(re-derive again)"]
    end
    subgraph Persistent["✅ Persistent Memory — Compound over time"]
        P1["Session 1\n→ writes memory"] --> P2["Session 2\n→ reads + updates memory"]
        P2 --> P3["Session 3\n→ builds on both"]
    end
    style P1 fill:#166534,color:#fff
    style P2 fill:#166534,color:#fff
    style P3 fill:#166534,color:#fff


Architecture options:


    

  • File-based — structured knowledge in files the agent reads on startup (CLAUDE.md, memory dirs). Simple, transparent, version-controllable.
    • 

  • Database-backed — conversation summaries, user preferences, learned facts. Better for high-volume structured data.
    • 

  • Vector stores — embed and retrieve relevant past interactions semantically. Good for fuzzy recall across large histories.
    • 

  • Hybrid — structured files for stable knowledge + vector search for dynamic recall. Most practical.
    • 



Real-world applications:


    

  • Stock market analysis agents tracking portfolio history across sessions
    • 

  • Development agents remembering codebase architecture (Claude Code’s memory system)
    • 

  • Research agents compounding knowledge over time (this wiki)
    • 



Why it works: Instead of re-deriving answers (error-prone), the agent verifies against accumulated, previously-validated knowledge. Each session builds on the last.


Pattern 4: Managed Agent Infrastructure (Not DIY Harnesses)


Anthropic’s Managed Agents (beta, 2026) represent a new architectural pattern: instead of building your own agent loop, you use managed cloud infrastructure that handles orchestration, sandboxing, and session persistence.


Core abstractions:


    

  • Agent — Configuration (model + system prompt + tools + MCP servers). Defines WHAT.
    • 

  • Environment — Cloud container with isolated execution. Defines WHERE.
    • 

  • Session — Running instance connecting agent to environment. Defines WHEN/HOW.
    • 

  • Events — Messages, tool calls, and results flowing through sessions. The protocol.
    • 



Why it improves reliability: Container isolation, built-in session persistence, event streaming, and rate limiting are handled by the platform — not the developer.


The open alternative: LangChain’s Deep Agents Deploy offers the same pattern — model-agnostic and MIT licensed — with memory in standard formats (AGENTS.md, skills files) that users own and can self-host.


The market is splitting: integrated platforms (Anthropic — battle-tested, locked-in) vs. open harnesses (LangChain — flexible, newer). Both validate Pattern 4’s core insight: stop building harnesses, start configuring agents.


Agents vs Workflows — The Decision That Matters Most


Before choosing a design pattern, choose the right orchestration style. Matt Pocock (aihero.dev) gives the clearest definition:





































DimensionWorkflowAgent
StepsPredetermined by developer codeChosen by LLM from available tools
StoppingCode decides when to stopLLM decides when it thinks it’s done
PredictabilityHigh — same path every timeLower — improvising toward goal
Best forClearly specified, repeatable tasksOpen-ended tasks where steps aren’t clear
Compounding errorBounded (known N steps)Unbounded (LLM chooses chain length)




Key insight (Matt Pocock): “Workflows are often unfairly maligned because they’re not as exciting and sexy as agents. You’ll often get better results from using a workflow than using an agent, as long as the task is clearly specified.”




This matches Anthropic’s core recommendation: “Avoid agentic systems if simple single-call optimization suffices.”

    Q1 -->|No| Q2{"Needs\nimprovisation?"}
    Q2 -->|Yes| A["🤖 Agent\n(LLM decides steps)"]
    Q2 -->|No| ST["Break into sub-tasks\n→ Workflow"]
    ST --> W

Workflows also enable parallelization: split input into N parts, process each independently, merge results. Pure workflow — no agent needed, dramatically more reliable.


Anthropic’s Agent Design Taxonomy


Building Effective Agents identifies six workflow patterns:










































PatternWhen to UseHow It Mitigates Compounding Error
Prompt ChainingTask decomposes into fixed subtasksProgrammatic “gates” validate each step
RoutingInputs need classificationReduces per-path step count via specialization
ParallelizationIndependent subtasksMultiple signals reduce single-point failure
Orchestrator-WorkersComplex tasks requiring dynamic decompositionBreaks long chains into shorter sub-chains
Evaluator-OptimizerClear evaluation criteria existIterative refinement with external validation
Autonomous AgentsOpen-ended problemsRequires sandboxing, guardrails, stopping conditions


Core recommendation: “Avoid agentic systems if simple single-call optimization suffices.”

This Wiki as Case Study


Karpathy's LLM Wiki gist — the original pattern this wiki implements


This project embodies all four patterns:


    

  • Knowledge compilation: Raw sources → synthesized wiki pages with cross-references. Errors are visible in compiled pages, not hidden in ephemeral chat.
    • 

  • Persistent memory: The wiki IS the memory — each session builds on previous work.
    • 

  • Tool search: As the wiki grows, load only relevant pages for each query.
    • 

  • Managed infrastructure: Claude Code handles session management, file access, and tool orchestration.
    • 





Key insight: The meta-pattern across all four approaches: move work out of the unreliable real-time LLM call and into reliable pre-computed infrastructure.
