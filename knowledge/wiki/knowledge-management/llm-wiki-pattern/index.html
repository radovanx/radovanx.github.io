The LLM Wiki Pattern — From Bookmarks to Knowledge Compilation

Andrej Karpathy’s pattern for persistent, compounding knowledge bases maintained by LLMs. The key insight: stop re-deriving answers from raw sources (RAG) and start compiling knowledge into a durable artifact (wiki) that grows richer with every source and question.

This wiki you’re reading is a direct implementation.

Why Not RAG?

RAG retrieves fragments at query time and re-derives connections every time. It’s stateless — the same question asked twice does the same work twice. The wiki pattern front-loads synthesis:

RAGKnowledge Compilation
Cross-referencesRe-derived each queryAlready built
ContradictionsInvisibleAlready flagged
SynthesisEphemeral (chat)Durable (wiki page)
Cost per queryHigh (retrieval + generation)Low (read a page)
Compounds over timeNoYes

“The wiki is a persistent, compounding artifact.”

Architecture

Three layers:

  1. Raw Sources (raw/) — immutable curated documents. Articles, bookmarks, notes. The LLM reads from them but never modifies them.
  2. The Wiki (wiki/) — LLM-generated markdown. Summaries, concept pages, synthesis pages that connect multiple sources. The LLM owns this layer entirely.
  3. Schema (CLAUDE.md) — tells the LLM how the wiki is structured, what conventions to follow, what workflows to run.

The critical distinction: raw sources are immutable truth. The wiki is the LLM’s interpretation, synthesis, and organization of that truth. Humans curate sources; the LLM writes the wiki.

Operations

OperationWhat it does
IngestProcess new sources: read, categorize, write summaries, merge with existing knowledge, update cross-references
QueryAnswer questions from wiki; good answers get filed back as new pages (explorations compound)
LintHealth check: contradictions, stale claims, orphan pages, missing links, knowledge gaps
PruneRemove outdated or irrelevant sources and their wiki pages
ArchiveHide articles from main view without deleting

The merge step during ingest is crucial. A new source about “AI reliability” shouldn’t create a new page — it should update the existing reliability page, flag contradictions, and potentially create a synthesis connecting it to other concepts.

Why It Works

The maintenance bottleneck in traditional wikis is human effort. LLMs eliminate this: “The wiki stays maintained because the cost of maintenance is near zero.” The LLM handles cross-references, consistency, contradiction detection — all the tedious bookkeeping humans abandon.

Practical Implementations

Obsidian + Claude Code (Nate Herk)

Combines Obsidian’s local-first markdown vault with Claude Code as the wiki maintainer:

  • Obsidian’s graph view visualizes knowledge connections
  • Backlinks and search provide human-friendly navigation
  • Local-first storage keeps everything in plain markdown
  • Claude Code’s file access enables automated maintenance

Self-Evolving Memory (Cole Medin)

Builds persistent memory for Claude Code using the knowledge base pattern:

  • File system access maintains a wiki as long-term memory
  • Conversation insights are auto-ingested back into the knowledge base
  • CLAUDE.md schema guides maintenance behavior
  • Each conversation compounds on previous knowledge

This Project

This Knowledge Wiki implements the pattern with:

  • Raindrop.io for source curation (bookmark → sync → ingest)
  • Claude Code as the LLM (reads CLAUDE.md, writes wiki pages)
  • Astro static site for browsing and searching
  • GitHub Actions for daily automated sync

Scaling Challenges

At ~100 sources, the pattern works with simple keyword search and full-context loading. Beyond that:

  • Vector search for semantic retrieval (find related pages even when keywords don’t match)
  • Tool search and persistent memory ([[ai-agents/ai-agent-reliability]]) for dynamic context loading and maintaining wiki state across sessions

Community Ecosystem

The pattern spawned multiple implementations: quicky-wiki, SwarmVault, grover, wikigen, wikimind. Extensions include confidence scoring, temporal tracking, graph-based retrieval, and interactive visualization.

Connection to Agent Reliability

The wiki pattern is itself a response to the [[ai-agents/ai-agent-reliability|LLM reliability problem]]. Instead of trusting an LLM to derive correct answers from raw sources in real-time (unreliable), you pre-compile knowledge into structured pages where errors are visible and correctable. It’s the “guardrail” approach applied to knowledge management.

The pattern isn’t unique to wikis. [[ai-security/glasswing-project|Project Glasswing]] applies the same principle to vulnerability detection: rather than asking an AI to reason through attack surfaces on-demand, Anthropic pre-compiled pattern libraries of vulnerability signatures that Claude Mythos can scan against. Pre-synthesis beats real-time derivation in both knowledge management and security scanning.

Sustaining a compounding knowledge system also requires the disposition described in [[productivity/self-belief-strategic-optimism]] — trust that each session deposits something lasting, even when individual results feel thin. The value of this wiki isn’t any one ingest; it’s the accumulated cross-references and synthesis that make the 50th question faster and richer than the first.
