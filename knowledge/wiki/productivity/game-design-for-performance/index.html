Game Design for Performance — Lessons from RollerCoaster Tycoon

Sources

  • The Gold Standard of Optimization: A Look Under the Hood of RollerCoaster Tycoon (Lars Tofus, March 2026) — deep dive into RCT’s optimization techniques: Assembly, data type selection, bit shifting, pathfinding design, crowd simulation

RollerCoaster Tycoon (1999) by Chris Sawyer simulates full theme parks with thousands of agents on hardware from 1999 — without dropping frames. The OpenRCT2 re-implementation (100% compatible, fully reverse-engineered by fans) gives us access to the techniques without access to the original source code.

OpenRCT2 — open-source reimplementation of RollerCoaster Tycoon, fully reverse-engineered by fans

RCT is often cited as the last major game written almost entirely in Assembly. But Assembly alone doesn’t explain the performance — the real insight is deeper.

Key insight: The biggest performance gains in RCT came not from micro-optimization of code, but from designing the game itself around performance constraints. When the game designer and programmer are the same person, the design space changes fundamentally.

Technique 1: Data Types Sized to Expected Range

Most codebases reach for int64 or double by default. RCT used exactly as many bytes as each value required:

ValueBytesWhy
Overall park value4 bytesLarge numbers possible
Shop item price1 byteSmall number range only
Other money valuesMatched to max expectedPer-variable analysis

This reduces memory bandwidth and cache pressure. OpenRCT2 removed this optimization — normalized all money to 8-byte variables — because modern CPUs handle uniform types without performance penalty.

The lesson: Right-sizing data types matters most when memory bandwidth and cache lines are the bottleneck. At 1999 hardware specs, every byte counted.

Technique 2: Formulas Designed for Bit-Shifting

// In OpenRCT2's source (preserved from original):
NewValue = OldValue << 2;  // multiply by 4 (power of 2 → bit shift)
NewValue = OldValue >> 3;  // divide by 8 (power of 2 → bit shift)

// What most modern code would write:
NewValue = OldValue * 4;
NewValue = OldValue / 8;

Bit shifting is faster than general multiplication on older CPUs. But the surprising part: RCT’s game design formulas were written around powers of 2. Ride rating calculations, ticket prices, happiness curves — wherever possible, numbers like 4, 8, 16, 32 appear instead of 7, 9, 11.

This required the game designer to agree to change “9.5” to “8” because 8 is a number the CPU prefers. In any organization where designers and engineers are separate roles, this conversation can’t happen. Sawyer’s dual role made it possible.

Modern compilers now handle this automatically most of the time — but not reliably (MSVC with optimizations off, older clang versions). OpenRCT2 preserves the bit-shift syntax as insurance.

Technique 3: Pathfinding Designed as a Feature

The obvious pathfinding design for a theme park game:

  1. Guest decides which ride to visit
  2. Guest finds path to that ride

This is expensive: pathfinding for thousands of agents simultaneously is the classic O(n × pathfinding_cost) problem.

RCT’s actual design:

  • Guests walk through the park almost randomly, choosing direction at junctions with minimal rules to avoid dead ends
  • They discover rides by stumbling upon them, not by seeking them
  • No proactive destination planning — guests react to what they encounter

Traditional pathfinding still runs in specific cases (mechanic to broken ride, guest to park exit), but with a hard depth limit:

ActorPathfinder junction limit
Default guest5 junctions
Guest with park map7 junctions (map item increases this)
Mechanic8 junctions (mechanics are more important)

When the pathfinder hits its limit without finding a path, it fails and returns “not found.” The guest thinks “I can’t find the exit” — which appears in their thought bubbles.

Key insight: A technical limitation (pathfinder depth cap) became a gameplay feature (guests complaining about not finding the exit). The constraint that was forced by performance requirements was later justified by design. The limitation was invisible to players; the emergent behavior felt authentic.

This could not happen in a typical game studio where engineers and designers negotiate requirements. Sawyer designed the constraint and the feature simultaneously.

Technique 4: Crowd Simulation Without Collision

Simulating 2,000+ agents with collision detection and avoidance in 1999 is infeasible:

Naive approach: O(n²) collision checks per frame
1,000 guests: 1,000,000 checks per frame

RCT’s approach: Guests don’t collide. Thousands can share a single path tile. Instead, each guest tracks how many other guests are nearby. If too crowded, happiness drops and they complain.

Player experience before:  plan layout to avoid guest congestion
RCT's actual simulation:   plan layout to avoid guest unhappiness from proximity counts
Player experience after:   identical — they still need to manage crowd flow

The player outcome is the same. The simulation cost is orders of magnitude lower. This is not a bug — it’s a performance-first design that produces the same observed behavior.

The Meta-Pattern: Design Space ≠ Technical Space

The deepest lesson from RCT is architectural:

    Designer["🎨 Game Designer\n(what to build)"] -.->|"separate teams"| Engineer["⚙️ Engineer\n(how to build)"]
    Engineer -->|"performance impossible"| Push["push back, negotiate,\nor skip constraint"]
    
    Sawyer["👤 Chris Sawyer\n(designer + engineer)"] -->|"simultaneous"| Design["design WITH performance\nconstraints"]
    Design --> Feature["technical limit\nbecomes design feature"]

What separate roles cannot produce:

  • Formulas that account for binary arithmetic constraints
  • Pathfinding systems designed as both features and optimizations
  • Crowd simulations where the “correct” UX outcome is achieved via a simplified model

What dual roles enable:

  • Design decisions informed by runtime cost
  • Technical limits rationalized as features
  • Performance optimization at the specification level, not the implementation level

This pattern appears in other contexts: when the person who specifies behavior also bears the cost of implementing it, specifications tend to be leaner and more performance-conscious.

Relevance to Modern Systems

The principles generalize beyond 1999 game dev:

RCT techniqueModern analog
Data types sized to rangeDatabase column types; serialization format choice
Bit-shift-friendly formulasAlgorithm design with power-of-2 data structures

Random walk over full pathfindingApproximate algorithms over exact solutions
Proximity count over collision detectionApproximate nearest neighbor over exact search
Crowd sharing tilesAgent simulation without individual collision; aggregate stats instead

The pathfinding depth cap maps directly to LLM agent reliability: capping the number of tool calls an agent makes is the same tradeoff — accept some failures, preserve system performance and predictability.

// TypeScript: depth-capped BFS — mirrors RCT's junction limit (5 for guests, 8 for mechanics)
// Same tradeoff: bounded worst-case cost at the price of some unsolvable paths.

function depthCappedBFS<T>(
  start: T,
  isGoal: (node: T) => boolean,
  neighbors: (node: T) => T[],
  maxDepth: number // RCT: 5 guest, 7 with map, 8 mechanic
): T[] | null {
  const queue: Array<{ node: T; path: T[]; depth: number }> = [
    { node: start, path: [start], depth: 0 },
  ];
  const visited = new Set<string>([JSON.stringify(start)]);

  while (queue.length) {
    const { node, path, depth } = queue.shift()!;
    if (isGoal(node)) return path;
    if (depth >= maxDepth) continue; // hard cap — "I can't find the exit" (visible feature)

    for (const n of neighbors(node)) {
      const key = JSON.stringify(n);
      if (!visited.has(key)) {
        visited.add(key);
        queue.push({ node: n, path: [...path, n], depth: depth + 1 });
      }
    }
  }
  return null; // path not found — the technical limit becomes a gameplay signal
}

// Agent turn limit: same pattern, different domain
// Each LLM tool call = one "junction"; cap at N to prevent runaway sessions
async function agentWithTurnLimit(
  task: string,
  callLLM: (ctx: string) => Promise<{ done: boolean; output: string; context: string }>,
  maxTurns = 20 // RCT mechanic junction limit equivalent
): Promise<string> {
  let context = task;
  for (let turn = 0; turn < maxTurns; turn++) {
    const result = await callLLM(context);
    if (result.done) return result.output;
    context = result.context;
  }
  return `[Incomplete — stopped after ${maxTurns} turns]`; // fail visibly, not silently
  // Output: bounded cost; same as RCT guest thought "Can't find the exit"
}


<?php
// PHP: depth-capped BFS + proximity-count approximation (RCT's O(n) crowd simulation)

function depthCappedBFS(
    mixed $start,
    callable $isGoal,
    callable $neighbors,
    int $maxDepth = 8
): ?array {
    $queue   = [['node' => $start, 'path' => [$start], 'depth' => 0]];
    $visited = [serialize($start) => true];

    while (!empty($queue)) {
        ['node' => $node, 'path' => $path, 'depth' => $depth] = array_shift($queue);
        if ($isGoal($node)) return $path;
        if ($depth >= $maxDepth) continue; // junction limit reached

        foreach ($neighbors($node) as $neighbor) {
            $key = serialize($neighbor);
            if (!isset($visited[$key])) {
                $visited[$key] = true;
                $queue[] = ['node' => $neighbor, 'path' => [...$path, $neighbor], 'depth' => $depth + 1];
            }
        }
    }
    return null; // Output: null — path not found within depth limit
}

// Proximity count instead of collision detection — O(n) not O(n²)
// Each guest tracks nearby_count; happiness drops if too crowded.
// Same player-visible outcome as collision detection; orders of magnitude cheaper.
function updateProximityCounts(array &$agents, int $tileSize = 1): void
{
    // Reset
    foreach ($agents as &$a) { $a['nearby'] = 0; }

    // O(n²) scan — but n is small per tile-bucket in a real implementation
    for ($i = 0; $i < count($agents); $i++) {
        for ($j = $i + 1; $j < count($agents); $j++) {
            $dx = abs($agents[$i]['x'] - $agents[$j]['x']);
            $dy = abs($agents[$i]['y'] - $agents[$j]['y']);
            if ($dx <= $tileSize && $dy <= $tileSize) {
                $agents[$i]['nearby']++;
                $agents[$j]['nearby']++;
            }
        }
    }
    // Happiness penalty applied from nearby count — not individual collision resolution
    // Output: same crowd-flow player experience; no per-agent collision geometry needed
}
